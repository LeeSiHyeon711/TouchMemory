  - `lifespan`에서 `init_db(config.db_path)` 호출(deprecated `@app.on_event` 미사용)
  - `get_db()` FastAPI 의존성 — 요청마다 `get_conn` 열고 응답 후(예외 포함) `finally: conn.close()`로 커넥션 누수 방지

  - `_safe_event()` — 자동 기록(register/view/complete)은 try/except로 감싸 best-effort 처리(7절), 실패해도 본 요청은 성공 응답
  - 엔드포인트: `POST /items`(register 기록) / `GET /items`(project·status·today 쿼리, view 기록; `today=true`는 `build_today_digest` 결과 id집합으로 필터해 digest 로직 재사용, repository.py는 수정하지 않음) / `GET /items/{id}`(404) / `PATCH /items/{id}`(is_done false→true 전환 시만 complete 기록) / `DELETE /items/{id}` / `GET /digest/today` / `POST /events`(notify 등록, channel 무조건 `system` 강제) / `GET·PATCH /settings`(settings 테이블 전체 조회는 repository에 전체조회 함수가 없어 `conn.execute` 직접 사용, PATCH는 키별로 기존 `repository.set_setting` UPSERT 재사용)
- `requirements.txt` — `fastapi>=0.111,<1.0`, `uvicorn[standard]>=0.30,<1.0` 추가
- `.env.example` — `API_HOST`/`API_PORT` 추가 (실제 소비는 FEAT-06 run.sh 몫, 현재는 uvicorn 커맨드라인 `--port`로 직접 테스트)

2. `DB_PATH=data/touchmemory_test.db uvicorn touchmemory.api.server:app --port 8790` 기동 후:
   - `list`(빈 상태) → "(항목 없음)", exit 0
   - `add "CLI 테스트" --project TouchMemory --remind daily` → "등록 완료: #1", exit 0
   - `add "두번째 항목" --project TouchMemory --remind once --date 2026-07-10 --persistent` → "등록 완료: #2", exit 0
   - `list` → 표 형태(ID/내용/프로젝트/상태/리마인드)로 두 항목 정상 출력, `once(2026-07-10)(지속)` 표시 확인
   - `list --status todo` → 두 항목 모두 노출(둘 다 미완료) 확인
   - `edit 1 --content "수정됨" --persistent on` → "수정 완료: #1", exit 0 → 재조회 시 내용/persistent 실제 반영 확인(완료조건: "edit로 텍스트/지속값이 실제로 변경됨")
   - `edit 1`(옵션 없음) → "수정할 옵션이 없습니다..." + exit 1 (API 호출 전 클라이언트 사전 차단)
   - `edit 999 --content x`(존재하지 않는 id) → API 404 → "해당 항목 없음" + exit 1
   - `add "날짜없음" --remind once`(--date 누락) → API 400 detail("remind_date...")이 그대로 출력 + exit 1
   - `delete 1` → "삭제 완료: #1", exit 0 → `list`에서 사라짐 확인(완료조건: "delete로 항목이 제거됨")
   - `delete 999`(존재하지 않는 id) → 서버가 404 대신 `{"deleted": false}`를 200으로 반환하는 기존 동작(server.py 확인, FEAT-02 범위라 변경 안 함)이라 CLI가 그 값을 보고 "해당 항목 없음" + exit 1로 동일하게 처리
   - `API_BASE_URL=http://127.0.0.1:1 list`(연결 불가 포트) → "데이터 서버에 연결할 수 없습니다..." + exit 2
3. **기본값 폴백 검증**: 서버를 포트 8787(fallback 기본값)로 재기동 후 `unset API_BASE_URL` 상태로 `list`/`add` 실행 → 정상 동작 확인(완료조건: "API_BASE_URL 미설정 상태에서도 기본값으로 동작")
4. `sqlite3 data/touchmemory_test.db "SELECT event_type, channel, item_id FROM usage_events"` → 모든 행 `channel=claude_code`로 기록됨 확인(완료조건: "모든 호출이 X-Channel: claude_code 헤더로 API 경유")
5. Discord `/기억조회` 교차 확인은 이번 세션에서 미실시(DISCORD_TOKEN 없음) — 다만 CLI와 Discord 봇이 동일 중앙 API·동일 DB를 경유하는 구조이고(server.py 라우트가 채널 구분 없이 동일 테이블을 다룸), usage_events에 channel만 다르게 기록되는 것을 확인했으므로 데이터 공유 자체는 구조적으로 보장됨. 실제 Discord 슬래시커맨드로의 육안 교차 확인은 사람 수동 테스트 단계 권장
6. 테스트 후 `data/`, `.venv/`, `__pycache__/` 모두 삭제, uvicorn 프로세스 종료 — `git status --short` 결과 신규 `src/touchmemory/cli/`와 `빌드노트.md` 수정만 존재 확인
