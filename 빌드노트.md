# 빌드노트 — 이슈 #2 (FEAT-02 중앙 API 서버)

## 진행 상태: 완료 (commit 가능 상태, commit/push는 repo-manager 담당)

## 구현 파일
- `src/touchmemory/digest.py` — 신규 작성 완료 (`DigestGroup`/`TodayDigest` Pydantic 모델, `build_today_digest(conn, today)` — (A)once만기·(B)daily·(C)persistent_reminder·(D)전날 미완료 none, 4개 조건 UNION 후 id 기준 dict로 중복 제거 → project_tag별 그룹핑. **그룹 D는 `substr(created_at,1,10) < today` 사용, `date()` 미사용**)
- `src/touchmemory/api/__init__.py` — 신규 작성 완료 (패키지 마커)
- `src/touchmemory/api/server.py` — 신규 작성 완료
  - `lifespan`에서 `init_db(config.db_path)` 호출(deprecated `@app.on_event` 미사용)
  - `get_db()` FastAPI 의존성 — 요청마다 `get_conn` 열고 응답 후(예외 포함) `finally: conn.close()`로 커넥션 누수 방지
  - `_today()` — `datetime.now(ZoneInfo(config.tz)).date().isoformat()`로 TZ 로컬 날짜 계산 후 `build_today_digest`에 전달
  - `_channel(request)` — `X-Channel` 헤더 파싱(기본 `system`, 알 수 없는 값도 `system`으로 폴백)
  - `_validate_once()` — `remind_type='once'`인데 `remind_date` 없음/형식(`YYYY-MM-DD`) 오류 시 `HTTPException(400)`. POST는 요청 바디 기준, PATCH는 기존 항목값과 patch를 병합한 최종값 기준으로 검증(부분 업데이트로 remind_date가 사라지는 경우도 방지)
  - `_safe_event()` — 자동 기록(register/view/complete)은 try/except로 감싸 best-effort 처리(7절), 실패해도 본 요청은 성공 응답
  - 엔드포인트: `POST /items`(register 기록) / `GET /items`(project·status·today 쿼리, view 기록; `today=true`는 `build_today_digest` 결과 id집합으로 필터해 digest 로직 재사용, repository.py는 수정하지 않음) / `GET /items/{id}`(404) / `PATCH /items/{id}`(is_done false→true 전환 시만 complete 기록) / `DELETE /items/{id}` / `GET /digest/today` / `POST /events`(notify 등록, channel 무조건 `system` 강제) / `GET·PATCH /settings`(settings 테이블 전체 조회는 repository에 전체조회 함수가 없어 `conn.execute` 직접 사용, PATCH는 키별로 기존 `repository.set_setting` UPSERT 재사용)
- `requirements.txt` — `fastapi>=0.111,<1.0`, `uvicorn[standard]>=0.30,<1.0` 추가
- `.env.example` — `API_HOST`/`API_PORT` 추가 (실제 소비는 FEAT-06 run.sh 몫, 현재는 uvicorn 커맨드라인 `--port`로 직접 테스트)

## 설계 범위 밖이라 손대지 않은 것
- `src/touchmemory/repository.py`는 FEAT-02 "수정 예상 파일" 목록에 없어 수정하지 않음. `GET /items?today=true` 필터·`GET/PATCH /settings` 전체조회는 server.py 안에서 `build_today_digest` 재사용 또는 `conn.execute` 직접 호출로 처리(기존 repository 함수 시그니처·동작 변경 없음).

## 테스트 결과 (모두 통과, `.venv`+`data/`는 테스트 후 삭제)
1. `python3 -m venv .venv && pip install -r requirements.txt` → fastapi/uvicorn 설치 성공
2. `uvicorn touchmemory.api.server:app --port 8787` 기동 → `GET /docs` 200 확인 (lifespan에서 init_db 실행되어 `data/touchmemory.db` 자동 생성)
3. `POST /items`(remind_type=daily) → id 반환 확인 → `GET /items`에 등장 → `GET /digest/today`에서 B그룹으로 등장
4. 그룹 D: `created_at`을 어제 로컬 날짜(`2026-07-04T10:00:00+09:00`)로 직접 삽입한 remind_type=none·미완료 항목 → `/digest/today`에 D그룹으로 등장. `POST /items`로 당일 등록한 none 항목은 미등장 확인
5. **그룹 D 타임존 경계**: `created_at='2026-07-05T00:30:00+09:00'`(오늘 로컬 새벽, UTC로는 전날 15:30)로 삽입한 remind_type=none·미완료 항목이 `/digest/today`(today=2026-07-05)에서 **미등장** 확인 — `substr(created_at,1,10)` 접두 비교로 `date()`의 UTC 오프셋 오판을 피함
6. `PATCH /items/1`로 `is_done=true, persistent_reminder=true` 전환 후 `/digest/today` → 완료됐지만 C그룹으로 여전히 등장 확인
7. `once` 검증: `remind_date` 없이 `remind_type=once` POST → 400, 형식오류(`2026/07/05`) → 400, 정상 형식(`2026-07-05`) → 201 성공 확인
8. `POST /events`(`X-Channel: discord` 헤더 + 바디 `channel:discord` 모두 지정)로 notify 전송 → `usage_events` 조회 시 `channel=system`으로 강제 기록됨 확인(헤더·바디 모두 무시)
9. `sqlite3 usage_events` 조회로 `register`(claude_code/system 채널 구분)·`view`·`complete`·`notify` 이벤트가 각각 대응 접근마다 누적됨을 확인
10. `GET /items?today=true` → digest 대상 id 집합(B/D/C 그룹 항목)만 필터되어 반환됨 확인
11. `GET/PATCH /settings` → PATCH `{"digest_hour":"09"}` UPSERT 후 GET에서 동일 값 조회 확인
12. `GET /items/999`, `PATCH /items/999`(Content-Type: application/json 헤더로) → 둘 다 404 확인
13. `DELETE /items/{id}` → `{"deleted": true}` 확인
14. 테스트 후 `data/`, `.venv/`, `__pycache__/` 모두 삭제 — 저장소는 소스 코드만 남은 상태
