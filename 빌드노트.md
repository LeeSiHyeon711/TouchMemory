  - `_safe_event()` — 자동 기록(register/view/complete)은 try/except로 감싸 best-effort 처리(7절), 실패해도 본 요청은 성공 응답
  - 엔드포인트: `POST /items`(register 기록) / `GET /items`(project·status·today 쿼리, view 기록; `today=true`는 `build_today_digest` 결과 id집합으로 필터해 digest 로직 재사용, repository.py는 수정하지 않음) / `GET /items/{id}`(404) / `PATCH /items/{id}`(is_done false→true 전환 시만 complete 기록) / `DELETE /items/{id}` / `GET /digest/today` / `POST /events`(notify 등록, channel 무조건 `system` 강제) / `GET·PATCH /settings`(settings 테이블 전체 조회는 repository에 전체조회 함수가 없어 `conn.execute` 직접 사용, PATCH는 키별로 기존 `repository.set_setting` UPSERT 재사용)
- `requirements.txt` — `fastapi>=0.111,<1.0`, `uvicorn[standard]>=0.30,<1.0` 추가
- `.env.example` — `API_HOST`/`API_PORT` 추가 (실제 소비는 FEAT-06 run.sh 몫, 현재는 uvicorn 커맨드라인 `--port`로 직접 테스트)

## 이슈 #7 버그 수정 — `run.sh` PYTHONPATH 미설정으로 클린 환경 기동 실패

### 문제 (검수리포트.md 2절·9절)
- 클린 `.venv`에서 README ②③④ 절차 그대로 `./run.sh` 실행 시 API 서버가 `ModuleNotFoundError: No module named 'touchmemory'`로 기동 실패(100% 재현). `src/touchmemory/` src-layout인데 패키지 설치 스텝(`pyproject.toml`/`pip install -e .`)도 없고 `run.sh`도 `PYTHONPATH`를 export하지 않아 발생. builder가 각 FEAT 테스트 때마다 `PYTHONPATH=src`를 수동 지정해 우회해왔을 뿐 스크립트에는 반영되지 않았던 것이 원인.

### 수정 내용 (최소 수정, 범위 확대 없음)
- `run.sh`의 `.env` 로드 블록 직후에 한 줄 추가: `export PYTHONPATH="$(pwd)/src:${PYTHONPATH:-}"`
  - 스크립트 4행에서 이미 `cd "$(dirname "${BASH_SOURCE[0]}")"`로 스크립트 위치로 이동해 있으므로 `$(pwd)`가 곧 프로젝트 루트(`05-개발/`)와 같다. `${PYTHONPATH:-}` 폴백으로 기존 값이 있으면 보존.
- `.env`/`.env.example`에는 `PYTHONPATH` 키를 추가하지 않았다(설계서 8-6 "Python은 os.environ만 읽고 `.env` 로딩은 run.sh 책임" 원칙 유지 — export는 스크립트 자체 책임으로 둠).
- `pyproject.toml`/`pip install -e .` 같은 패키징 전환은 하지 않았다(검수리포트가 대안으로 언급했으나 이번엔 가장 좁은 수정만 적용).
- README 대조 결과: README ②③④ 어디에도 "PYTHONPATH를 직접 설정하라"는 안내가 없었다(그 자체가 이번 버그의 일부였음). 이번 수정으로 `run.sh`가 자동으로 처리하므로 README에 별도 안내를 추가할 필요는 없다고 판단해 README는 수정하지 않았다. (공방장 확인용 대조 결과만 보고)

### 테스트 방법 및 결과
1. 기존 `.venv`/`data`/`logs`/`backup` 없는 상태에서 `python3 -m venv /tmp/tm_clean_venv_test` → 신규 클린 가상환경 생성
2. `source /tmp/tm_clean_venv_test/bin/activate && pip install -r requirements.txt` → 정상 설치 확인
3. 수정된 `./run.sh`를 백그라운드로 실행 → `curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:8787/docs` → **200 확인**(`ModuleNotFoundError` 재현 안 됨, `logs/api.log`에 uvicorn 정상 기동 로그 확인)
4. 봇은 이 세션 로컬 환경의 CA 인증서 문제(`SSLCertVerificationError: unable to get local issuer certificate`, discord.com 접속 시)로 즉시 종료됨 — 이는 이 세션 환경 문제이며 이번 PYTHONPATH 수정과 무관(기존 검수리포트 9절에서도 동일 문제를 "코드 결함 아님"으로 별도 분류함). API 기동 성공 여부만으로 이번 수정을 판정.
5. 테스트 후 프로세스 정리(`pkill`) + 임시 `.venv`(`/tmp/tm_clean_venv_test`) 삭제. `git status --short` → `run.sh`만 수정된 상태로 정리됨 확인(테스트로 생성된 `data/`·`logs/`·`backup/`은 `.gitignore` 대상이라 노출 안 됨).

## 이슈 #8 UX 표시명 개선 — `type:chore` (표시명만, 내부 enum/DB/API/로직 변경 없음)

### 배경
사람 수동 테스트 중 `remind_type`/`persistent_reminder`의 화면 표시명이 직관적이지 않다는 지적(특히 `once`가 "한 번만"으로 오해됨). 공방장 승인된 표시명 4종을 Discord 봇·CLI·README에 반영.

### 확정 표시명
`none`→"내일부터 다시 보기", `once`→"지정일부터 완료까지 보기", `daily`→"매일 다시 보기", `persistent_reminder=true`→"완료해도 계속 보기"

### 수정 내용 (표시 문구만, 내부 값·로직 무변경)
- `src/touchmemory/bot/client.py`
  - `/기억등록`의 `리마인드` 파라미터에 `@app_commands.choices(...)` 데코레이터 추가(표시명 3종 드롭다운, `value`는 기존과 동일하게 `"none"/"once"/"daily"` 문자열 — API 호출·`_REMIND_TYPES` 체크·`remind_type=리마인드` 전달 로직은 무변경, `if 리마인드 not in _REMIND_TYPES` 방어 체크도 안전망으로 그대로 유지)
  - `@app_commands.describe`의 `리마인드`/`지속` 설명문을 표시명 기준으로 재작성(`지속`은 "완료해도 계속 보기 여부(기본 거짓)"로, 기존에 빠져있던 "완료 후에도 유지된다"는 핵심을 담음)
  - 리마인드 값 오류 메시지를 표시명 기준으로 재작성
  - `/기억완료`의 지속 리마인드 안내문을 "(완료해도 계속 보기 설정된 항목입니다)"로 다듬음
- `src/touchmemory/cli/main.py`
  - `add`/`edit`의 `--remind` **choices(`none/once/daily`)는 무변경**(스크립트 호환성 유지), `help=` 문자열에만 표시명 병기
  - `--persistent` help를 "완료해도 계속 보기 여부"로 개선
  - `render_items()` 표 렌더링 라벨만 교체: `once(날짜)`→`지정일부터완료까지(날짜)`, `daily`→`매일`, `(지속)`→`(완료해도계속)`, `-`는 그대로. **비교 로직(`item.get("remind_type")`이 여전히 `"once"/"daily"/"none"` 문자열인지 확인하는 부분)은 무변경**
- `README.md`: "리마인드(`remind_type`) 의미" 섹션 시작 부분(⑤ 아래)에 내부값↔표시명 대응표 추가. 기존 상세 설명 문단은 이미 기술적으로 정확해 그대로 유지

### 테스트 방법 및 결과 (모두 통과)
1. `python3 -m venv` 임시 venv 생성 후 `pip install -r requirements.txt` 성공(discord.py/httpx 등 정상 설치)
2. `DISCORD_TOKEN=dummy PYTHONPATH=src python3 -c "import touchmemory.bot.client"` → 임포트 오류 없음 확인
3. `tree.get_command('기억등록')._params['리마인드'].choices` 조회 → `[('내일부터 다시 보기','none'), ('지정일부터 완료까지 보기','once'), ('매일 다시 보기','daily')]` 정확히 노출 확인. `describe` 문구도 의도한 대로 출력 확인
4. `python -m touchmemory.cli add --help` / `edit --help` → `--remind {none,once,daily}`(choices 그대로) + help 문구에 표시명 병기 확인, `--persistent` help 문구 갱신 확인
5. `render_items()`에 once(날짜+지속)/daily/none(완료) 3개 항목을 넣어 직접 호출 → `지정일부터완료까지(2026-07-10)(완료해도계속)` / `매일` / `-` 정확히 출력 확인(내부 `remind_type` 비교 로직은 그대로이므로 화면 라벨만 바뀜을 확인)
6. API 서버를 띄우지 않고 순수 문자열/파서 단위 테스트만 진행했으므로 등록/조회 API 호출 값 자체는 코드 diff상 `remind_type=리마인드`(문자열 그대로 전달)·`patch["remind_type"] = args.remind` 등 기존 로직 변경 없음을 직접 확인(동작 값 변경 없음)
7. 테스트 후 임시 venv 삭제, `git status --short` → 의도한 3개 파일(`README.md`, `src/touchmemory/bot/client.py`, `src/touchmemory/cli/main.py`)만 수정됨 확인
